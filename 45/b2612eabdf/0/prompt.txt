<result>Task: explain the motivation for the qualifying thresholds (≥6 sessions, ≥2 held-out sessions, ≥8 held-out turns).

Each threshold guards a distinct requirement for a user to be a usable eval point:

- **≥6 sessions** — enough material to *build a profile from*. The profile is distilled from the user's earlier (train) sessions; one or two sessions give too thin a behavioral record to characterize. ~6 is the floor at which `prepare_data.py` yields a meaningful digest.

- **≥2 held-out sessions** — makes the per-user *temporal* split real. The split is chronological (earlier sessions → profile, later sessions → scored), so a user needs sessions reserved on the held-out side that weren't used to build their profile. This is the within-user no-leakage guard.

- **≥8 held-out turns** — keeps each per-user CondAgree estimate from being pure noise. CondAgree is a per-user macro (each user contributes one fraction, averaged across users); with only 2–3 turns a user's fraction swings wildly (0/3, 3/3) and adds variance without signal. 8 is a minimum for a marginally stable per-user rate.

**Why these specific (relaxed) values:** they were lowered from a stricter earlier bar (≥10 sessions / ≥4 held-out sessions / more turns) that yielded only ~20 qualifying users total — too few to meet the "≥20 users in *each* of test and val" requirement. Relaxing to (6, 2, 8) widens the pool to ~40 so test and val each get 20. The trade-off favors more users because the cross-user macro CI (the actual ± error bars) tightens with user count, while the resulting noisier per-user estimates average out.

Caveat: these are pragmatic round thresholds picked to satisfy the ≥20-per-split constraint on a small pool, not products of a formal power analysis.
